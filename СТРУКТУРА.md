# Структура проекта «magazin»

Статический веб‑сайт для учёта магазина (приход товара, продажи, возвраты,
расходы, склад, отчёты). Фронтенд — обычные HTML/CSS/JS без фреймворков и
сборщиков. Бэкенд — Supabase (база данных PostgreSQL, аутентификация, хранилище
фото и Edge Functions). Хостинг — Vercel.

- **Версия:** 1.1.0
- **Языки интерфейса:** русский (`ru`) и таджикский (`tj`)
- **Роли пользователей:** `admin` (полный доступ) и `manager` (только **добавление**
  новых записей; правка/удаление и удаление товаров — только `admin`). Разграничение
  ролей и журнал изменений проверяются **на сервере** в Edge Function `save-store`.
- **CSP:** `script-src 'self'` (без `unsafe-inline`); весь UI — на делегировании
  событий через `data-act` (inline-обработчиков в разметке нет).
- **Supabase project ref:** `jqzzedaihqnqcjyxspky` (проект «magazin»)

## Фронтенд

### `index.html` (~102 строки)
Единственная страница приложения. Содержит:
- **Экран авторизации** (`#authPage`): формы входа, восстановления пароля и
  установки нового пароля.
- **Оболочку приложения** (`#appShell`): шапку (логотип, переключатель языка
  РУ/ТЈ, email пользователя, кнопка обновления, статус синхронизации, кнопки
  «Создать пользователя» и «Выйти»), навигацию `#nav` и `<main>` с восемью
  секциями-страницами.
- **Модальные окна**: создание пользователя (`#userModal`) и точка монтирования
  для модалок редактирования (`#editModalMount`).
- В `<head>` задан строгий **Content-Security-Policy** (`script-src 'self'`, без
  `unsafe-inline`). Кнопки/поля используют `data-act` / `data-input-act` /
  `data-backdrop` — обработчики навешиваются делегированием (см. `app.js`).
- В конце подключаются скрипты в строгом порядке:
  `config.js → sync.js → auth.js → products.js → operations.js → ui-tabs.js → app.js`.

**Страницы (вкладки):** `dashboard` (Главная), `arrived` (Товар пришёл),
`sales` (Продал), `returns` (Возврат), `expenses` (Расход), `stock` (Склад/Анбор),
`report` (Отчёт), `history` (История).

### `styles.css` (~1375 строк)
Все стили: тема, шапка, карточки, формы, таблицы (включая компактные
`compactTable` для телефонов), модальные окна, графики, адаптивная вёрстка
вплоть до экранов шириной 380px.

### `config.js` / `config.example.js` (по 4 строки)
Задают `window.STORE_SUPABASE = { url, anonKey }`.
- `config.js` — реальные ключи, **не коммитится** (есть в `.gitignore`); на
  Vercel генерируется автоматически из переменных окружения.
- `config.example.js` — шаблон-заглушка для локальной разработки.

### `app.js` (~1640 строк) — ядро
Самый большой модуль. Отвечает за:
- **Константы localStorage**: `DB_KEY`, `SESSION_KEY`, `CHANGES_KEY`,
  `ACTIVE_TAB_KEY`, `LANG_KEY`.
- **Словари переводов** `T` (ru/tj) и функции `tr()`, `setLang()`.
- **Глобальное состояние**: `db`, `currentUser`, `currentRole`, `changeHistory`,
  выбранный период, состояние модалок, поиск, сортировка склада и т.д.
- **Описание данных**: коллекции `products, arrivals, sales, returns, expenses,
  writeoffs` + «надгробия» удалённых записей (tombstones).
- **Утилиты**: форматирование денег и дат, диапазоны периодов (`range`,
  `inPeriod`), расчёты (`totals`, `cashFlow`, `payableTotal`, `receivableTotal`,
  `salesByDay`), графики продаж, экспорт в JSON/CSV, экранирование HTML
  (`escapeHtml`, в т.ч. одинарной кавычки). `newId()` — уникальный числовой id
  (мс·1000 + random) против коллизий при оффлайн-создании на разных устройствах.
- **Endpoints Supabase** (REST, auth, functions, storage) и заголовки запросов.
- **UI**: формы авторизации/восстановления, уведомления (`showNotice`), тосты,
  главный цикл рендеринга страниц.
- **Делегирование событий** (в конце файла): единый обработчик на `document` +
  реестры `CLICK_ACTIONS` / `INPUT_ACTIONS` / `BLUR_ACTIONS`. Заменяет inline
  `onclick`/`oninput`, что позволило убрать `unsafe-inline` из CSP. Любой новый
  интерактивный элемент помечается `data-act` и регистрируется здесь.

### `sync.js` (~364 строки)
Слой данных и синхронизации:
- `readLocalDb()` / `normalizeDb()` — чтение и строгая нормализация локальной БД
  из `localStorage` (очистка значений, ограничение длины, типы полей).
- Слияние локальных и серверных данных, обработка удалений (tombstones),
  отправка состояния через Edge Function `save-store`, обновление из Supabase.

### `auth.js` (~218 строк)
Аутентификация и сессии:
- `setAuthView()` — переключение между экраном входа и приложением.
- `authRequest()` — запросы к Supabase Auth (`/token`, `/recover`, `/user`).
- Управление сессией, автообновление токена (`sessionRefreshTimer`), загрузка
  роли пользователя, восстановление и смена пароля.

### `products.js` (~362 строки)
Работа с товарами:
- `makeSku()` — генерация артикула, `stockClass()` — цветовая индикация остатка
  (красный ≤1, оранжевый 2–4, зелёный ≥5).
- Создание товаров (доступно и менеджеру), расчёт остатков, история по товару,
  рендер таблицы склада. **Редактирование и удаление товара — только `admin`**
  (гард в `openProductEditModal`, кнопки скрыты у менеджера).

### `operations.js` (~411 строк)
Операции магазина:
- Поле выбора даты операции (`opDateField`, нельзя выбрать будущее).
- Пагинация таблиц (`TABLE_PAGE = 20`, «Показать ещё»).
- Сортировка записей (новые сверху по дате операции, затем по id).
- Формы и обработка прихода, продажи, возврата, расхода и списания (добавление —
  доступно менеджеру).
- `rowActions()` — кнопки правки/удаления строки; **показываются только `admin`**.
  Гарды в `deleteRecord()` и `openEditModal()` дублируют проверку на клиенте.

### `ui-tabs.js` (~144 строки)
Навигация по вкладкам: список вкладок (`getTabs`), рендер `#nav`,
переключение активной страницы, запоминание выбранной вкладки в `localStorage`,
обновление переключателя языка.

## Сборка и деплой

### `scripts/build.mjs` (~28 строк)
Скрипт сборки (Node, ESM):
1. Создаёт папку `dist/`.
2. Копирует туда `index.html`, все `.js` и `styles.css`.
3. Генерирует `dist/config.js` из переменных окружения
   `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` (или `SUPABASE_*`).
4. Предупреждает в консоль, если переменные не заданы.

### `package.json`
`name: magazin`, `version: 1.1.0`, `private: true`, единственный скрипт —
`"build": "node scripts/build.mjs"`.

### `vercel.json`
- `buildCommand: npm run build`, `outputDirectory: dist`.
- Security-заголовки для всех маршрутов: строгий **CSP** (`script-src 'self'`, без
  `unsafe-inline`; `style-src` оставлен с `unsafe-inline` ради инлайновых стилей),
  `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`,
  `Permissions-Policy`. CSP продублирован в `<meta>` внутри `index.html`.

## Бэкенд (Supabase)

### `supabase-schema.sql` (~148 строк)
Полная схема БД, применяется в SQL Editor Supabase:
- **Таблицы:**
  - `store_state` — единое состояние магазина в JSONB (id = `'main'`).
  - `user_roles` — роль каждого пользователя (`admin` / `manager`).
  - `store_changes` — журнал изменений (кто, когда, какое действие).
- **Storage:** приватный bucket `product-photos` (лимит 1 МБ, только webp/jpeg/png).
- **GRANT‑ы** для ролей `authenticated` и `service_role`.
- **Row Level Security**: чтение состояния, своей роли, журнала и фото — только
  для пользователей с ролью `admin`/`manager`; запись фото — в папку
  пользователя по `auth.uid()`.
- В комментарии — инструкция, как назначить первому пользователю роль `admin`.

### `supabase/config.toml`
Настройка Edge Functions: `save-store` и `create-user` с `verify_jwt = false`
(JWT проверяется внутри самих функций).

### `supabase/functions/save-store/index.ts` (~477 строк)
Edge Function (Deno) для **сохранения** состояния магазина и **единственная точка
записи** в `store_state` (через `service_role`). Здесь же — серверная защита:
- **Разграничение ролей** в `mergeStore(base, incoming, isAdmin)`: для админа —
  last-write-wins с учётом удалений; для менеджера — база источник истины,
  принимаются только записи с **новыми** ключами (правки/удаления игнорируются).
- **Достоверный журнал** `describeChange(before, after)`: `action`/`details` для
  `store_changes` формируются из реальной серверной разницы, а не из текста
  клиента (подделать нельзя). Пустые изменения не логируются.
- **Лимит тела** `readJsonCapped()`: потоковое чтение с обрывом на 2 МБ (не
  доверяет `content-length`).
- **Очистка фото**: осиротевшие `.webp` (товар удалён/фото заменено) удаляются из
  bucket `product-photos` через `service_role`.
- **CORS**: разрешены только источники `*.vercel.app` и `localhost`; для точного
  ограничения — секрет функции `ALLOWED_ORIGINS` (домены через запятую).
- Также: проверка метода POST, защита от гонки записи (по `updated_at`).

### `supabase/functions/create-user/index.ts`
Edge Function (Deno) для **создания пользователя** через `service_role`. Доступно
только администратору; при ошибке записи роли новый Auth‑пользователь не остаётся
без роли (ошибка возвращается явно). CORS — как в `save-store` (только свои
источники / `ALLOWED_ORIGINS`).

### `supabase/.temp/linked-project.json`
Служебный файл Supabase CLI с привязкой к удалённому проекту (`ref: jqzzedaihqnqcjyxspky`).

## Поток данных (кратко)

1. Пользователь входит через **Supabase Auth** (`auth.js`).
2. Загружается его **роль** из `user_roles`. Интерфейс адаптируется (у менеджера
   скрыты кнопки правки/удаления и создания пользователя), но **решающая проверка
   роли — на сервере** в `save-store` (UI не является защитой).
3. Все данные хранятся локально в `localStorage` и синхронизируются с
   `store_state` (JSONB) через Edge Function **`save-store`** (`sync.js`).
   Менеджер может лишь добавлять новые записи.
4. Журнал **`store_changes`** (вкладка «История») заполняется сервером из реальной
   разницы состояния — клиентский текст изменения не используется.
5. Фото товаров грузятся в приватный bucket **`product-photos`**; неиспользуемые
   фото удаляются сервером при сохранении.
6. Сборка (`build.mjs`) копирует файлы в `dist/` и подставляет ключи Supabase из
   переменных окружения Vercel.
